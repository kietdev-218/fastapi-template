# Standard API Response (BE ↔ FE Integration Guide)

Objective: All endpoints return the **same envelope structure**, allowing FE to write a single handler layer
(interceptor, error handler, type/interface) instead of handling each API individually.

---

## 1. General Envelope

```json
{
    "success": true,
    "code": 0,
    "message": "OK",
    "data": {},
    "meta": {},
    "errors": null,
    "timestamp": "2026-07-18T10:00:00Z",
    "traceId": "a1b2c3d4-uuid"
}
```

| Field       | Type              | Required | Description                                                                                                                                                                                                        |
| ----------- | ----------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `success`   | boolean           | ✅       | `true`/`false` – FE only needs to check this field to branch happy/error paths.                                                                                                                                    |
| `code`      | **int**           | ✅       | Integer business identification code (not HTTP status), used by FE for switch-case logic (e.g., showing different modals). `0` = success, other numbers = specific error/situation codes (see table in section 4). |
| `message`   | string            | ✅       | User-readable message (should support i18n keys instead of hardcoded text for multi-language).                                                                                                                     |
| `data`      | object/array/null | ✅       | The actual payload. Always use the same field name `data`, do not rename it per API (avoid `user`, `list`, `result`...).                                                                                           |
| `meta`      | object            | optional | Auxiliary info: pagination, applied filters, version...                                                                                                                                                            |
| `errors`    | array/null        | optional | Detailed error info (which field failed validation) – null when success.                                                                                                                                           |
| `timestamp` | string (ISO 8601) | ✅       | Time the server finished processing.                                                                                                                                                                               |
| `traceId`   | string            | ✅       | Used for logging/debugging, maps to server logs. Automatically generated by OpenTelemetry for distributed tracing.                                                                                                 |

## 2. Success Response

### 2.1. Single object

```json
{
    "success": true,
    "code": 0,
    "message": "Information retrieved successfully",
    "data": {
        "id": "usr_001",
        "name": "Nguyen Van A",
        "email": "a@example.com"
    },
    "meta": null,
    "errors": null,
    "timestamp": "2026-07-18T10:00:00Z",
    "traceId": "req-8f2e1a"
}
```

### 2.2. List + Pagination

```json
{
    "success": true,
    "code": 0,
    "message": "List retrieved successfully",
    "data": [
        { "id": "1", "name": "Item A" },
        { "id": "2", "name": "Item B" }
    ],
    "meta": {
        "pagination": {
            "page": 1,
            "pageSize": 20,
            "totalItems": 132,
            "totalPages": 7,
            "hasNext": true,
            "hasPrev": false
        }
    },
    "errors": null,
    "timestamp": "2026-07-18T10:00:00Z",
    "traceId": "req-9a11cd"
}
```

> Pagination convention: Always use `page` (starting from 1) + `pageSize`, do not mix with `limit/offset` in other APIs
> so FE can reuse pagination components.

## 3. Error Response

### 3.1. General Business Error

```json
{
    "success": false,
    "code": 20404,
    "message": "User not found",
    "data": null,
    "meta": null,
    "errors": null,
    "timestamp": "2026-07-18T10:00:00Z",
    "traceId": "req-4c77ee"
}
```

### 3.2. Form Validation Error (multiple fields)

```json
{
    "success": false,
    "code": 80400,
    "message": "Invalid data",
    "data": null,
    "meta": null,
    "errors": [
        { "field": "email", "message": "Invalid email format" },
        { "field": "password", "message": "Password must be at least 8 characters" }
    ],
    "timestamp": "2026-07-18T10:00:00Z",
    "traceId": "req-1123ab"
}
```

> Thanks to `errors[].field`, FE can map errors directly to each input (react-hook-form, formik...) without parsing
> messages by string.

## 4. `code` (int) Conventions by Domain

Since `code` is now an integer, a **fixed numbering structure** is required to avoid duplication and allow FE/BE to look
up meanings without asking each time a new code is added.

### 4.1. Number Structure: `[2-digit domain][1-digit HTTP group][2-digit sequence]`

```
  2       0       404
domain  http-grp  sequence
```

- **First 2 digits**: business domain (00 = general/system, 10 = auth, 15 = permission, 20 = user, 30 = order, 40 =
  payment...)
- **Next 1 digit**: corresponding HTTP error group (0 = 2xx success, 4 = 4xx client error, 5 = 5xx server error)
- **Last 2 digits**: specific sequence in that domain (freely assigned, incrementing)

> The number of digits can be customized based on system scale, as long as the **length is fixed** and recorded in a
> shared table (source of truth), preventing each service from assigning numbers arbitrarily.

### 4.2. Domain Table

| Domain code | Meaning                                                                                                                                                  |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `00`        | System / Common Infrastructure layer error, not related to any business logic (Server error, timeout, rate limit, maintenance mode, service unavailable) |
| `10`        | Auth / Identity Authentication & session (Login, logout, refresh token, token expiration, incorrect password, OTP, 2FA)                                  |
| `15`        | Authorization / Permission (Permission assignment, RBAC, insufficient role permissions, locked resource, missing scope)                                  |
| `20`        | User (users, profiles)                                                                                                                                   |
| `30`        | Order (orders)                                                                                                                                           |
| `40`        | Payment (payments)                                                                                                                                       |
| `50`        | Product / Catalog                                                                                                                                        |
| `60`        | Search (General search: Query error, index not ready, empty results)                                                                                     |
| `70`        | Integration / Third-party (Call to external service: Third-party API timeout, webhook error, OAuth with Google/Facebook fail)                            |
| `80`        | Validation / Input (General input data error: Incorrect format, missing required fields, duplicate data)                                                 |

### 4.3. Example Code Table

| `code`  | HTTP Status | Meaning                                                 |
| ------- | ----------- | ------------------------------------------------------- |
| `0`     | 200         | Success                                                 |
| `1`     | 201         | Created successfully                                    |
| `00400` | 400         | General bad request                                     |
| `00401` | 401         | Not logged in                                           |
| `00403` | 403         | Forbidden (no permission)                               |
| `00429` | 429         | Rate limit                                              |
| `00500` | 500         | System error                                            |
| `10401` | 401         | Token expired                                           |
| `10402` | 401         | Invalid token                                           |
| `20404` | 404         | User not found                                          |
| `20409` | 409         | Email already exists                                    |
| `30404` | 404         | Order not found                                         |
| `30422` | 422         | Product out of stock                                    |
| `40422` | 422         | Insufficient balance                                    |
| `70500` | 500         | Payment gateway error/timeout (Integration/Third-party) |
| `80400` | 400         | Validation error (invalid input data)                   |

> This table should be generated/defined in **a single place** (a shared constants file, or generated from
> OpenAPI/Swagger spec) and imported by both BE and FE, avoiding hardcoded scattered numbers that can easily go out of
> sync.

### 4.4. Notes When Using int Codes

- Because they are not self-describing strings (`USER_NOT_FOUND`), **`message` becomes more important** to display to
  the user — do not rely solely on `code` to infer content.
- A mapping file (JSON/TS enum) should be exported for FE to look up mnemonic names when debugging, for example:
    ```json
    { "0": "SUCCESS", "20404": "USER_NOT_FOUND", "40422": "INSUFFICIENT_BALANCE" }
    ```
- When adding new domains/codes: only **add**, do not reuse old "retired" numbers (deprecated codes keep their number,
  mark as deprecated in docs).

## 5. Naming Conventions & Data Types

- **camelCase** for all fields (do not mix `snake_case` and `camelCase` across APIs).
- Date and time: always ISO 8601 UTC (`2026-07-18T10:00:00Z`), FE will convert to local timezone.
- ID: always use `string` (even if the DB uses integers) to avoid large number precision loss in JS
  (`Number.MAX_SAFE_INTEGER`).
- Currency: return as integers (smallest unit, e.g., cents) with a `currency` field, avoid floats.
    ```json
    { "amount": 150000, "currency": "VND" }
    ```
- Enums in `data` (resource status, e.g., order status): should still return **strings** (`"ACTIVE"`, `"PENDING"`) so FE
  doesn't need to look up a table. This is different from `code` in the envelope (section 4) — `code` uses int because
  it's only for technical switch-case logic, not direct display.
- Null fields vs non-existent fields: if an optional field has no value, **always return `null`**, do not completely
  omit the field from the JSON (helps FE use fixed TypeScript interfaces, avoids random `undefined` checks).

---

## 6. Versioning & Backwards Compatibility

- Version via URL: `/api/v1/users`, `/api/v2/users` — do not version via header (easy for FE to forget to set).
- When adding new fields: always make them **optional, do not break old fields** (additive change).
- When needing to change an old field: create a new API version, do not modify old fields in the current version.

## 7. Corresponding TypeScript Interface Example (For FE use)

```typescript
// Should define shared enum/mapping, generated from the code table in section 4
enum ApiCode {
    SUCCESS = 0,
    CREATED = 1,
    BAD_REQUEST = 400,
    UNAUTHORIZED = 401,
    VALIDATION_ERROR = 80400,
    USER_NOT_FOUND = 20404,
    TOKEN_EXPIRED = 10401,
    INSUFFICIENT_BALANCE = 40422,
}

interface ApiResponse<T> {
    success: boolean;
    code: number;
    message: string;
    data: T | null;
    meta: {
        pagination?: {
            page: number;
            pageSize: number;
            totalItems: number;
            totalPages: number;
            hasNext: boolean;
            hasPrev: boolean;
        };
        [key: string]: unknown;
    } | null;
    errors: { field: string; message: string }[] | null;
    timestamp: string;
    traceId: string;
}
```

FE only needs 1 shared axios/fetch interceptor:

```typescript
if (!response.data.success) {
    switch (response.data.code) {
        case ApiCode.VALIDATION_ERROR:
            // map response.data.errors to form
            break;
        case ApiCode.UNAUTHORIZED:
        case ApiCode.TOKEN_EXPIRED:
            // redirect to login
            break;
        default:
        // show toast based on response.data.message
    }
}
```

## 8. Checklist When BE Builds New APIs

- [ ] Returns correct envelope (`success`, `code`, `message`, `data`, `meta`, `errors`, `timestamp`, `traceId`)
- [ ] `data` always has the same shape for the same resource whether in list or detail
- [ ] Includes `traceId` for log tracing
- [ ] Validation errors return `errors[]` mapped by field
- [ ] Does not change old fields when updating API within the same version
- [ ] `code` is an integer, matching the domain table defined in section 4 (do not assign random numbers)
- [ ] Enums/status in `data` return strings, not magic numbers
- [ ] IDs return strings
- [ ] New `code` is added to the shared mapping file (constants/OpenAPI spec), not just hardcoded individually in BE
